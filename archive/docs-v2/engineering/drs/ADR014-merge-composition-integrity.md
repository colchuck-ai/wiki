# ADR014 - Merge composition integrity

Make the `merge` composition [ADR011](ADR011-concept-verb-surface.md) introduced — `revise(into, source=from)` then `delete(from, replacement=into)` — hold together as a composition on the two points ADR011/ADR012 left implicit: (1) express its review-candidate suppression through an explicit `content_preserved` parameter on `delete` that `merge` passes, so the difference from a supersession `delete` is a truthful argument value rather than a hidden merge-only code path; and (2) give the composition an **atomicity contract** — `merge` commits as a single unit or not at all — so a failure of the `delete` step after `revise` has already folded content in cannot leave a silent duplicate. Extends [ADR011](ADR011-concept-verb-surface.md) (verb atomicity, now stated to compose) and [ADR012](ADR012-repair-to-successor-link-model.md) (review-candidate suppression, now given a mechanism).

## Context

[ADR011](ADR011-concept-verb-surface.md) defined `merge(from, into)` as a **named composition** of primitives — `revise(into, source=from)` then `delete(from, replacement=into)` — explicitly *not* a fourth write primitive, "no new authoring or removal machinery." C003's success criterion states it as a *pure composition*: a `merge` "produces exactly the state of `revise(into, from)` followed by `delete(from, replacement=into)` — no separate merge-only code path — verifiable by comparing a `merge` against the two calls run in sequence." Two facts recorded elsewhere quietly break that as written.

**The suppression mechanism is unspecified.** [ADR012](ADR012-repair-to-successor-link-model.md) decided that `review_candidates` "track content removal, not link repair": a `delete` surfaces them (the content is gone; a referrer's prose may no longer hold), and a `merge` does not (its `revise` step folded the content into the target the links now point at). But a `merge`'s removal step *is* a `delete(from, replacement=into)` — the exact signature a steward uses to **supersede** one concept with another (O003-R003's "delete with a replacement"). A supersession `delete` surfaces review candidates (the replacement is a different concept, not guaranteed to carry what each referrer relied on); a `merge`'s `delete` suppresses them. The signature `delete(concept_id, [replacement_concept_id])` carries nothing that distinguishes the two, so the suppression can only be implemented as a merge-only branch inside `delete` — the very thing "no merge-only code path" forbids — and the criterion's own test (compare `merge` against a standalone `revise`+`delete`) would then *fail*, because the standalone `delete` would surface the candidates the `merge` suppressed.

**The composition has no atomicity contract.** ADR011 guarantees every *verb* is "atomic including its effects." But `merge` is two verbs across two components — C003's `revise`, then C008's `delete` — and nothing says the *pair* is atomic. If `delete` fails after `revise` has already folded `from`'s content into `into` and committed, the corpus is left with the content in `into` **and** the original `from` still present: a silent duplicate — precisely O003-RSK003 (redundant capture), the risk `merge` exists to resolve, now manufactured by the tool. Per-verb atomicity does not compose on its own.

Constraints are unchanged: the corpus is plain-text and git-native (a change is a diffable, mergeable set of file edits); repair is a mechanical link-syntax edit through C004; the envelope is low-volume, single-steward. Both gaps are about the *composition*, not the primitives — `revise` and `delete` each remain correct in isolation.

## Options

**How to express a `merge`'s `delete` step vs. a supersession `delete` (Problem A):**

- **A truthful `content_preserved` parameter on `delete` (chosen)**: extend the signature to `delete(concept_id, [replacement_concept_id], content_preserved=false)`. The flag states a fact about the corpus — *the departing concept's content survives at the replacement* — which `merge` can assert because its preceding `revise` made it true, and which a bare supersession `delete` leaves false (the replacement is a different concept). `delete` derives review-candidate suppression from the flag: suppress when `content_preserved`, surface otherwise. `merge` is then `revise(into, from)` + `delete(from, replacement=into, content_preserved=true)` — no merge-only branch; the two behaviors are one code path parameterized by a truthful argument, and the pure-composition test holds when the same flag is passed.
- **An `is_merge` flag on `delete`**: simplest to wire, but it leaks the *composition* into the primitive — `delete` would branch on "am I being called by merge?" rather than on a fact about the content. It also mis-describes the supersession case (a steward who has genuinely reconciled content by hand before deleting could never say so). Rejected: `content_preserved` states the load-bearing fact directly and generalizes.
- **Compute review candidates in a separate read after `delete`**: keep `delete` purely mutation+repair; a follow-up query decides review candidacy. Rejected on ordering: review candidacy is a function of the concept's *inbound links*, which `delete` has already removed by the time it returns — the referrers must be captured *during* the deletion, so the computation genuinely belongs to `delete`. Splitting it would force the query to reconstruct pre-deletion state.

**How to guarantee the composition is all-or-nothing (Problem B):**

- **Composition atomicity — one commit or none (chosen)**: `merge`'s `revise` and `delete` steps participate in a **single transaction**; `merge` materializes as one atomic unit or not at all, and a failure of either step leaves the corpus byte-for-byte unchanged — no fold without removal, no removal without fold. In the git-native corpus this is realized as **one commit or none**: the steps stage their edits and effects (content, citations, link repair, recency, history, reindex) into one working-tree change that commits once; a mid-way failure aborts and restores the tree. This refines ADR011's per-verb atomicity into a rule that *composes*: a verb run standalone is its own atomic unit; a verb run inside a composition joins the composition's unit.
- **Accept the window; let Lint catch the duplicate (status quo by omission)**: run `revise` and `delete` as two independent atomic verbs (two commits) and rely on a later Lint/Survey pass to notice a `from` that should be gone. Rejected: it reintroduces exactly the correctness-by-convention footgun ADR011 removed for materialization ("index/recency drift eliminated by construction rather than caught by Lint"), and here the failure mode is worse than drift — it is a silent duplicate concept, the O003-RSK003 the verb is meant to prevent, left in the corpus until someone notices.
- **Make `merge` a fourth primitive with its own transactional body**: guarantees atomicity trivially by not being a composition. Rejected: it discards ADR011's decision that `merge` is sugar over `revise`+`delete`, reintroducing a third authoring/removal path to keep coherent. Composition atomicity gets the guarantee without the duplicated machinery.

## Decision

Adopt both refinements, together making `merge` a genuine, safe composition.

`delete` gains a third parameter, `content_preserved` (default `false`): `delete(concept_id, [replacement_concept_id], content_preserved=false)`. It asserts that the departing concept's content survives at `replacement_concept_id`. `delete` surfaces `review_candidates` when `content_preserved` is false (content removed) and suppresses them when it is true (content preserved at the successor). `merge` passes `content_preserved=true`; a supersession `delete` (and every steward-facing bare `delete`) leaves it false. This replaces the implicit merge-detection ADR012 left unspecified: review-candidate suppression is now driven by a truthful argument, not by `delete` knowing it is inside a `merge`. C003's pure-composition criterion narrows accordingly — a `merge` yields the **same corpus bytes and the same repaired links** as `revise(into, from)` followed by `delete(from, replacement=into, content_preserved=true)`, and, passing that same flag, the same (empty) review report; there is still no merge-only code path.

`merge` is **atomic as a composition**: its `revise` and `delete` steps run in one transaction and materialize as a single unit or not at all — one git commit or none in the git-native corpus. A failure of either step leaves the corpus byte-for-byte unchanged: no content folded into `into` without `from` removed, and no `from` removed without its content folded in. This refines ADR011's per-verb atomicity so that atomicity **composes**: within `merge`, the `revise` and `delete` effects (recency, history, reindex, link repair) still all occur, but the commit is deferred to the composition boundary rather than taken twice.

Both are refinements, not reversals: ADR011's verb surface and ADR012's repair-to-successor rule and review-candidate-tracks-content-removal principle stand unchanged; this ADR supplies the mechanism (`content_preserved`) and the composition-atomicity contract they assumed but did not state.

## Consequences

- The pure-composition claim becomes true as written: with `content_preserved=true` passed to the standalone `delete`, a `merge` is byte-for-byte and report-for-report identical to `revise`+`delete`, so C003's verification test passes instead of failing. No merge-only code path is introduced.
- `delete`'s interface grows by one parameter, but stays deep: all repair and review-candidate machinery remains hidden, and the new flag is a single truthful boolean. A steward-facing bare `delete` never sets it; only the `merge` composition (and, in principle, a hand-reconciled supersession) does.
- The silent-duplicate failure mode is closed by construction: a `delete` failure inside `merge` rolls the fold back with it, so the corpus can never hold both the merged content and the original `from`. O003-RSK003 is prevented by the verb rather than re-created by it.
- Atomicity now has a stated composition rule. The cost is that a composition defers its constituent verbs' commits to the composition boundary — the verbs must be able to run their effects without committing when composed. This is a small addition to the write path, acceptable in the low-volume, single-steward envelope, and consistent with ADR011's intent that materialization never drift from content.
- A future composition of write verbs (should one arise) inherits the same rule — atomicity composes — rather than each new composition re-deciding it.
