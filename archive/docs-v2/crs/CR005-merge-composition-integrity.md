# CR005 - Merge composition integrity: content_preserved and atomicity

Updated C008 - Lifecycle & Retirement and C003 - Integration Authoring so the `merge` composition (`revise` + `delete`) holds together as a composition. `delete` gains a `content_preserved` parameter that drives review-candidate suppression; C003's pure-composition success criterion narrows to corpus bytes and repaired links; and `merge` carries an explicit all-or-nothing atomicity contract (one git commit or none).

## Change

Before: `merge(from, into)` was defined as `revise(into, source=from)` then `delete(from, replacement=into)`, with two gaps left implicit by [ADR011](../engineering/drs/ADR011-concept-verb-surface.md) and [ADR012](../engineering/drs/ADR012-repair-to-successor-link-model.md). (1) `delete(concept_id, [replacement_concept_id])` had no parameter distinguishing a `merge`'s content-preserving removal from a supersession `delete` — identical signature, opposite review-candidate behavior — so the specified suppression could only be a merge-only branch inside `delete`, and C003's own "pure composition, verifiable by comparing `merge` against a standalone `revise`+`delete`" test would have failed (the standalone `delete` surfaces the candidates `merge` suppresses). (2) `merge` spanned two verbs across two components with no composition-atomicity contract: a `delete` failure after `revise` had folded content into `into` would leave the content in `into` *and* the original `from` present — a silent duplicate (O003-RSK003), created by the tool.

After: `delete` becomes `delete(concept_id, [replacement_concept_id], content_preserved=false)`. The flag asserts that the departing concept's content survives at the replacement; `delete` suppresses `review_candidates` when it is true and surfaces them when false. `merge` passes `content_preserved=true`; a supersession `delete` and every steward-facing bare `delete` leave it false. C003's pure-composition criterion narrows: a `merge` yields the same corpus bytes and the same repaired links as `revise(into, from)` followed by `delete(from, replacement=into, content_preserved=true)`, and — passing that same flag — the same (empty) review report; there is still no merge-only code path. `merge` is now atomic **as a composition**: its `revise` and `delete` steps run in one transaction and materialize as a single unit or not at all (one git commit or none), so a failure of either leaves the corpus byte-for-byte unchanged. C008's Interfaces, Data model, Behavior, Edge cases, and Success criteria and C003's Interfaces, Behavior, and Success criteria are updated.

## Rationale

ADR011 named `merge` a *pure composition* with "no merge-only code path" and guaranteed each *verb* atomic "including effects"; ADR012 decided a `merge` suppresses the review candidates its `delete` step would otherwise raise. Both statements assumed a mechanism they did not supply. Without a parameter on `delete`, the suppression contradicts "no merge-only code path" and breaks the pure-composition test; without a composition-atomicity contract, per-verb atomicity does not compose, so a partial `merge` can manufacture the exact duplicate the verb exists to prevent. Expressing suppression through a truthful `content_preserved` flag keeps `merge` a genuine composition, and contracting `merge` as one-commit-or-none closes the silent-duplicate window by construction rather than leaving it for a later Lint/Survey pass — the same reasoning ADR011 used to make materialization a verb effect instead of a caller step. See [ADR014](../engineering/drs/ADR014-merge-composition-integrity.md) for the full option analysis, including why an `is_merge` flag and a separate post-`delete` review query were both rejected.

## Affects

- **C008 - Lifecycle & Retirement** — `delete` gains the resolve-unchanged `content_preserved` parameter; review-candidate suppression is driven by it; the `merge`-step edge case and the review-surfaced/suppressed success criterion are restated in its terms.
- **C003 - Integration Authoring** — `merge` passes `content_preserved=true` to its `delete` step; the pure-composition success criterion narrows to corpus bytes + repaired links (+ same-flag review report); the composition carries the one-commit-or-none atomicity contract.
- **O003-R003 - Duplication check** — served more safely: the in-corpus merge remediation can no longer half-complete into a duplicate, and O003-RSK003 (redundant capture) stays prevented by the verb rather than re-created by a partial one.
- **Wiki Curation Architecture** (`docs/engineering/README.md`) — the "Write verbs carry their own effects" principle gains a clause stating that atomicity composes: the `merge` composition commits as one unit or not at all.
- **ADR014** — records the decision and alternatives; extends **ADR011** (verb atomicity now composes) and **ADR012** (review-candidate suppression now expressed via `content_preserved`).
